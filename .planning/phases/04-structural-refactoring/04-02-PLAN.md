**Create `packages/core/src/scheduler/services/lifecycle-manager.ts`:**

The LifecycleManager follows the identical pattern to existing services (JobScheduler, JobManager, etc.):
- Receives `SchedulerContext` via constructor
- Marked `@internal`
- Manages: poll interval, heartbeat interval, cleanup interval, and cleanupJobs()

**LifecycleManager responsibilities (extracted from Monque):**

1. **Timer IDs** — `pollIntervalId`, `heartbeatIntervalId`, `cleanupIntervalId` move from Monque to LifecycleManager as private fields.

2. **`startTimers(callbacks)` method** — Replaces the timer setup logic currently in `Monque.start()`:
   - Accepts callbacks: `{ poll: () => Promise<void>, updateHeartbeats: () => Promise<void> }` — these call through to `JobProcessor.poll()` and `JobProcessor.updateHeartbeats()` respectively
   - Sets up poll interval with error catch-and-emit
   - Sets up heartbeat interval with error catch-and-emit
   - Sets up cleanup interval if `options.jobRetention` configured (runs immediately then on interval)
   - Runs initial poll immediately
   - Uses `toError()` for error normalization (import from `@/shared`)

3. **`stopTimers()` method** — Replaces the interval-clearing logic currently in `Monque.stop()`:
   - Clears all three intervals
   - Sets interval IDs to null

4. **`cleanupJobs()` method** — Move the entire `cleanupJobs()` implementation from Monque (lines 397-429) with its JSDoc intact.

5. **DEFAULTS** — The `retentionInterval` default (3600_000) can be referenced. Either import the DEFAULTS constant or duplicate just the one value. Claude's discretion on approach.

**Update `packages/core/src/scheduler/services/types.ts`:**
- If LifecycleManager needs any additions to `SchedulerContext`, add them. Most likely no changes needed — it already has `collection`, `options`, `emit`, and `isRunning`.

**Update `packages/core/src/scheduler/services/index.ts`:**
- Add `export { LifecycleManager } from './lifecycle-manager.js';` to the services barrel.
- LifecycleManager is exported from the services barrel (internal) but NOT from the package public API (`src/index.ts`).

**Update `packages/core/src/scheduler/monque.ts`:**

1. Add `LifecycleManager` to the import from `./services/index.js`
2. Add `private _lifecycleManager: LifecycleManager | null = null;` alongside other service fields
3. Add private getter:
   ```typescript
   /** @throws {ConnectionError} if not initialized */
   private get lifecycleManager(): LifecycleManager {
     if (!this._lifecycleManager) {
       throw new ConnectionError('Monque not initialized. Call initialize() first.');
     }
     return this._lifecycleManager;
   }
   ```
4. In `initialize()`: add `this._lifecycleManager = new LifecycleManager(ctx);`
5. **Remove from Monque:**
   - `pollIntervalId`, `heartbeatIntervalId`, `cleanupIntervalId` fields
   - `cleanupJobs()` method entirely
   - Timer setup code from `start()` (the setInterval calls for poll, heartbeat, cleanup, and immediate poll)
   - Timer clearing code from `stop()` (the clearInterval calls)
6. **Replace in `start()`:**
   ```typescript
   // After change stream setup
   this.lifecycleManager.startTimers({
     poll: () => this.processor.poll(),
     updateHeartbeats: () => this.processor.updateHeartbeats(),
   });
   ```
   Keep the `this.isRunning = true;` and `this.changeStreamHandler.setup();` lines.
7. **Replace in `stop()`:**
   ```typescript
   // After stats cache clear
   this.lifecycleManager.stopTimers();
   // Keep: change stream close, active jobs wait, shutdown timeout logic
   ```
   Keep the change stream close, active jobs wait/timeout logic on Monque per CONTEXT.md.

**Do NOT change:**
- `register()` method (stays on Monque)
- Shutdown wait/orchestration logic (Promise.race between active job completion and timeout)
- Change stream setup (already delegated to ChangeStreamHandler)
- `createIndexes()`, `recoverStaleJobs()`, `checkInstanceCollision()` — stay on Monque
- Any public method signatures
- The public API barrel (`src/index.ts`)
