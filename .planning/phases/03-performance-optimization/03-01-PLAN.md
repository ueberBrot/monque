**In `packages/core/src/scheduler/services/job-manager.ts`:**

**cancelJobs():** Replace the entire cursor-based implementation with:
1. Build base query from `buildSelectorQuery(filter)`
2. Add status guard: merge `{ status: JobStatus.PENDING }` into the query (only pending jobs can be cancelled per existing logic). If filter already specifies status, intersect with PENDING — if filter status doesn't include PENDING, return `{ count: 0, errors: [] }` early.
3. Single `updateMany` call: `this.ctx.collection.updateMany(query, { $set: { status: JobStatus.CANCELLED, updatedAt: new Date() } })`
4. Get `modifiedCount` from result
5. Emit `jobs:cancelled` with `{ count: modifiedCount }` if count > 0
6. Return `{ count: modifiedCount, errors: [] }` — per user decision, errors array is always empty for bulk ops

**retryJobs():** Replace cursor-based implementation with:
1. Build base query from `buildSelectorQuery(filter)`
2. Add status guard: merge `{ status: { $in: [JobStatus.FAILED, JobStatus.CANCELLED] } }` into query. If filter already specifies status, intersect with [FAILED, CANCELLED] — if no overlap, return early.
3. Use **pipeline-style updateMany** (array as second argument) for staggered `nextRunAt`:
   ```typescript
   const spreadWindowMs = 30_000; // 30s max spread — per user decision, use fixed reasonable spread
   await this.ctx.collection.updateMany(query, [
     {
       $set: {
         status: JobStatus.PENDING,
         failCount: 0,
         nextRunAt: {
           $add: [new Date(), { $multiply: [{ $rand: {} }, spreadWindowMs] }]
         },
         updatedAt: new Date(),
       },
     },
     {
       $unset: ['failReason', 'lockedAt', 'claimedBy', 'lastHeartbeat', 'heartbeatInterval'],
     },
   ]);
   ```
   Note: `$rand` returns a random float [0, 1) per-document, so each job gets a unique stagger. The user's CONTEXT.md mentions dynamic spread `min(jobCount * 100ms, 30000ms)` but that requires a count query first (two round-trips). Per "Claude's Discretion" in CONTEXT.md, use a fixed 30s window — small batches naturally cluster near 0 while large batches spread across 30s. This is O(1) round-trips.
4. Get `modifiedCount` from result
5. Emit `jobs:retried` with `{ count: modifiedCount }` if count > 0
6. Return `{ count: modifiedCount, errors: [] }`

**Important implementation notes:**
- The `$unset` in pipeline-style uses array of field name strings, not the `{ field: '' }` syntax used in traditional updates
- `new Date()` in pipeline `$set` stages evaluates once at pipeline compilation time (server-side), which is fine for `updatedAt` but `nextRunAt` stagger needs `$add`/`$rand` per the pipeline above
- Import nothing new — `buildSelectorQuery` and `JobStatus` already imported
- Do NOT change `deleteJobs()` — it already uses `deleteMany` (O(1))
- Do NOT change single-job operations (`cancelJob`, `retryJob`) — per user decision they stay as findOneAndUpdate
- Keep the existing JSDoc on `cancelJobs()`/`retryJobs()` but update the descriptions to reflect the new behavior (no per-job errors, count-only events)

**Status filter intersection logic:**
For `cancelJobs`: if `filter.status` is provided and doesn't include 'pending', return `{ count: 0, errors: [] }`. If filter.status includes 'pending' along with others, narrow to just 'pending' in the query.
For `retryJobs`: if `filter.status` is provided and has no overlap with ['failed', 'cancelled'], return `{ count: 0, errors: [] }`. Otherwise, intersect.
Simplest approach: override the status in the query after `buildSelectorQuery()` — e.g., `query['status'] = JobStatus.PENDING` for cancel, `query['status'] = { $in: [JobStatus.FAILED, JobStatus.CANCELLED] }` for retry. This ensures the DB only touches valid-state jobs regardless of what the user passes in filter.status.
