    **1. Add `satisfies` guard** to `packages/core/src/jobs/document-to-persisted-job.ts`:

    The goal: if someone adds a new required field to `Job<T>` but forgets to add it to the mapper, TypeScript compilation fails.

    Approach: After building the `job` object, add a `satisfies` assertion that checks the return type matches `Required<Pick<PersistedJob<T>, ...>>` for required fields. The simplest approach:

    - Change the function to build the object with all required fields first (already done), then assign optional fields conditionally (already done)
    - At the end, before `return job`, add: `job satisfies PersistedJob<T>;` — this is already implicitly checked because `job` is typed as `PersistedJob<T>`

    Better approach for compile-time detection: Create a type helper that maps required keys of `Job`:

    ```typescript
    // At top of file or in types.ts
    type RequiredJobKeys = {
      [K in keyof Required<Job>]: Job extends Record<K, Job[K]> ? K : never;
    }[keyof Job];
    ```

    Actually, the simplest effective approach: Add a `satisfies` check on the initial object literal that includes all required fields. The object literal assigned to `const job` already has type `PersistedJob<T>`, but the `satisfies` operator provides an additional structural check:

    Change the mapper to use a mapped-type guard. Replace the current approach with:

    ```typescript
    // Exhaustiveness guard type — ensures every key of PersistedJob is accounted for.
    // If a new required field is added to Job, this will cause a compile error
    // because the returned object won't satisfy the mapped type.
    type _ExhaustivenessCheck<T> = {
      [K in keyof Required<PersistedJob<T>>]: PersistedJob<T>[K];
    };
    ```

    Then at the return statement: `return job satisfies _ExhaustivenessCheck<T>;`

    Wait — `satisfies` won't catch missing optional fields. The issue from CONCERNS.md is: "Every new field added to the `Job` interface must also be added to this mapping function. If a field is missed, it silently drops data."

    The REAL exhaustiveness guard: Make the mapper return type `Required<PersistedJob<T>>` won't work since optional fields may not be present.

    **Correct approach:** Use a type that enumerates ALL Job keys and a compile-time assertion that they're all handled. Create a `_MAPPER_HANDLED_KEYS` const array listing every key, then use a type-level check:

    ```typescript
    // Type-level exhaustiveness: if a new key is added to Job but not to this array,
    // TypeScript emits a compile error.
    const _HANDLED_KEYS = [
      '_id', 'name', 'data', 'status', 'nextRunAt', 'failCount', 'createdAt', 'updatedAt',
      'lockedAt', 'claimedBy', 'lastHeartbeat', 'heartbeatInterval', 'failReason', 'repeatInterval', 'uniqueKey',
    ] as const satisfies readonly (keyof PersistedJob)[];

    // Reverse check: ensure no Job key is missing from _HANDLED_KEYS
    type _AssertAllKeysHandled = Exclude<keyof PersistedJob, (typeof _HANDLED_KEYS)[number]> extends never
      ? true
      : { error: 'documentToPersistedJob is missing keys'; missing: Exclude<keyof PersistedJob, (typeof _HANDLED_KEYS)[number]> };
    const _exhaustivenessCheck: _AssertAllKeysHandled = true;
    ```

    This produces a compile error if any key of `PersistedJob` is NOT in `_HANDLED_KEYS`. The error message will even show which keys are missing.

    Place the `_HANDLED_KEYS` array and type assertions at the bottom of `document-to-persisted-job.ts`, before the closing of the file. Prefix with underscore to indicate internal. The `_exhaustivenessCheck` const will be tree-shaken in production.

    **Important:** Biome will flag unused variables. Mark with: `// biome-ignore lint/correctness/noUnusedVariables: compile-time exhaustiveness guard`
