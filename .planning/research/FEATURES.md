# Feature Landscape: Monque Hardening Milestone

**Domain:** MongoDB-backed job queue library (hardening pass)
**Researched:** 2026-02-27
**Competitors analyzed:** BullMQ v5, Agenda/Agendash, pg-boss, Graphile Worker

---

## Table Stakes

Features users expect from a mature job queue library. Missing = product feels incomplete or unreliable.

### TS-1: Bulk Operation Optimization (cancelJobs/retryJobs)

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                 |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why Expected**         | BullMQ has `retryJobs()`, `clean()`, and `addBulk()` as first-class batched operations. pg-boss leverages SQL `UPDATE ... WHERE` for O(1) bulk state changes. Agenda exposes bulk requeue/delete through Agendash. Current Monque implementation is O(n) `findOneAndUpdate` calls via cursor iteration — a known performance bottleneck for any non-trivial job count. |
| **Complexity**           | Medium                                                                                                                                                                                                                                                                                                                                                                 |
| **Current State**        | `cancelJobs()` and `retryJobs()` in `job-manager.ts` iterate cursor with individual `findOneAndUpdate` per doc (lines 265-406)                                                                                                                                                                                                                                         |
| **Recommended Approach** | Two-pass strategy: (1) `updateMany` for jobs already in valid source state (common path, O(1) round-trip), (2) collect IDs that raced into invalid state for error reporting. `deleteJobs()` already uses `deleteMany` — match that pattern.                                                                                                                           |
| **Evidence Confidence**  | HIGH — verified against BullMQ API docs (retryJobs, clean methods) and pg-boss SQL-based approach                                                                                                                                                                                                                                                                      |

### TS-2: Stats/Metrics Caching (getQueueStats TTL)

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why Expected**         | BullMQ provides `getJobCounts()`, `getMetrics()`, and `exportPrometheusMetrics()` — all backed by Redis data structures optimized for fast reads. pg-boss has `getQueueSize()` and `monitorStateIntervalSeconds` to control polling frequency. Agenda exposes `getRunningStats()`. None of these libraries run an uncached full-collection aggregation pipeline on every call. Monque's `getQueueStats()` runs a `$facet` aggregation with `maxTimeMS: 30000` on every invocation — expensive and unnecessary for stats that rarely need sub-second freshness. |
| **Complexity**           | Low                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **Current State**        | `job-query.ts` lines 290-410 — uncached aggregation pipeline per call                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Recommended Approach** | Simple TTL-based in-memory cache with configurable duration (default 5s). Cache key should incorporate the optional `name` filter. Pattern: store `{ data, expiresAt }`, return cached if `Date.now() < expiresAt`, else re-run pipeline. No external cache dependency needed.                                                                                                                                                                                                                                                                                 |
| **Evidence Confidence**  | HIGH — verified BullMQ has dedicated getJobCounts/getMetrics APIs; pg-boss has monitorStateIntervalSeconds                                                                                                                                                                                                                                                                                                                                                                                                                                                     |

### TS-3: Payload Size Validation (maxPayloadSize)

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                       |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why Expected**         | BullMQ provides `sizeLimit` on `BaseJobOptions` — per-job byte limit after JSON serialization. If exceeded, the job add throws. MongoDB has a 16MB BSON document limit, and large payloads degrade collection performance via index bloat and WiredTiger contention. The current Monque code accepts arbitrary `data: T` with zero validation (security concern from audit). |
| **Complexity**           | Low                                                                                                                                                                                                                                                                                                                                                                          |
| **Current State**        | No validation — `enqueue()` and `schedule()` in `job-scheduler.ts` accept anything                                                                                                                                                                                                                                                                                           |
| **Recommended Approach** | Optional `maxPayloadSize` in `MonqueOptions` (bytes, applied globally). Check `JSON.stringify(data).length` against limit in `enqueue()`/`schedule()`/`now()` before insertion. Throw a new `PayloadTooLargeError` (extends `MonqueError`). No per-job override needed for this milestone — global is sufficient and simpler.                                                |
| **Evidence Confidence**  | HIGH — BullMQ `sizeLimit` on BaseJobOptions confirmed in official API docs                                                                                                                                                                                                                                                                                                   |

### TS-4: Instance ID Collision Detection (schedulerInstanceId)

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Why Expected**         | Distributed queue systems universally require worker identity guarantees. BullMQ uses Redis-backed worker tracking (getWorkers/getWorkersCount). pg-boss tracks worker instances internally and emits `wip` events. Graphile Worker uses PostgreSQL advisory locks for singleton guarantees. Monque defaults to `randomUUID()` (safe) but allows user override — two instances with the same ID would corrupt job claims and heartbeats. |
| **Complexity**           | Low                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Current State**        | `schedulerInstanceId` defaulting to `randomUUID()` with user override, no collision check                                                                                                                                                                                                                                                                                                                                                |
| **Recommended Approach** | On `initialize()`, query for any `processing` jobs with `claimedBy: thisInstanceId`. If found (and not our own from stale recovery), emit a warning event or throw `ConnectionError`. Lightweight — single query, runs once at startup. Do NOT use a persistent lock/heartbeat document — that adds complexity beyond what's needed for collision detection.                                                                             |
| **Evidence Confidence**  | MEDIUM — pattern is standard across distributed systems; specific implementation validated against BullMQ getWorkers and pg-boss wip event                                                                                                                                                                                                                                                                                               |

### TS-5: Test Coverage for Existing Features

| Aspect                   | Detail                                                                                                                                                                                           |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Why Expected**         | Every mature library has thorough test coverage for its public API. Missing tests for existing features are reliability gaps, not new features.                                                  |
| **Complexity**           | Low-Medium                                                                                                                                                                                       |
| **Current State**        | Three specific gaps identified in audit: (1) `MonqueModule.registerJobs()` unit tests in tsed, (2) `getQueueStats` aggregation timeout path test, (3) cleanup/retention concurrent instance test |
| **Recommended Approach** | Write targeted tests for each gap. No source changes needed for items 2-3. Item 1 requires mocking Ts.ED DI internals.                                                                           |
| **Evidence Confidence**  | HIGH — gaps directly identified in codebase audit                                                                                                                                                |

## Differentiators

Features that set Monque apart. Not expected, but valued by power users.

### D-1: Monque Class Size Reduction (Facade Decomposition)

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Value Proposition**    | BullMQ decomposes aggressively: `QueueBase` → `QueueGetters` → `Queue` inheritance chain, plus separate `Worker`, `QueueEvents`, and `FlowProducer` classes. Each class has a focused responsibility. Monque's `Monque` class is 1311 lines (400 logic + 900 JSDoc). The audit identified this as a maintenance burden — every API change requires updating JSDoc in two places (service + facade). Reducing this improves contributor DX and reduces documentation drift. |
| **Complexity**           | Medium                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Current State**        | Single facade class with duplicated JSDoc and embedded timer/lifecycle management                                                                                                                                                                                                                                                                                                                                                                                          |
| **Recommended Approach** | Two options, non-breaking: (A) Replace duplicated JSDoc with `@inheritdoc` or `@see` tags referencing service-level docs — reduces ~500 lines of duplication with zero logic changes. (B) Extract timer/interval management into a `LifecycleManager` internal service — cleaner separation but more refactoring. **Recommend option A first** (low risk, high impact on line count), then evaluate if B is needed.                                                        |
| **Evidence Confidence**  | HIGH — BullMQ class hierarchy confirmed from official API docs (Queue extends QueueGetters extends QueueBase)                                                                                                                                                                                                                                                                                                                                                              |
| **Notes**                | BullMQ's approach of separate Queue/Worker/QueueEvents classes is a different architecture — Monque's unified facade is a deliberate design choice. Don't break the facade pattern; just reduce internal bloat.                                                                                                                                                                                                                                                            |

### D-2: Document-to-Type Mapping Robustness (documentToPersistedJob)

| Aspect                   | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Value Proposition**    | Graphile Worker types payloads as `unknown` by default and encourages runtime validation (runtypes, zod). BullMQ stores jobs as Redis hashes with typed Job class wrappers. Monque's manual field-by-field mapping in `documentToPersistedJob` is fragile — adding a Job field without updating the mapper silently drops data. A more robust approach prevents silent data loss.                                                                                                                                                                                                                                           |
| **Complexity**           | Medium                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Current State**        | Manual field mapping in `monque.ts` lines 1246-1282 (already extracted to `jobs/` per PR #189)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **Recommended Approach** | Three progressive options: (A) **Exhaustive round-trip test** — a test that creates a job with ALL fields populated, round-trips through the mapper, and asserts every field survives. Catches regressions immediately. (B) **Spread-based mapper** — `{ ...doc, _id: doc._id }` with explicit field renames only where MongoDB field names differ from TS types. Automatically picks up new fields. (C) **Schema-driven mapper** — derive mapping from a Zod schema or const type map. Most robust but highest effort. **Recommend A + B together** — the test catches regressions, the spread-based mapper prevents them. |
| **Evidence Confidence**  | MEDIUM — pattern comparison is sound; specific mapper implementation is Monque-specific                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |

## Anti-Features

Features to explicitly NOT build in this hardening milestone.

| Anti-Feature                                                  | Why Avoid                                                                                                                                                                                     | What to Do Instead                                                                               |
| ------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **Per-job sizeLimit override**                                | BullMQ supports per-job `sizeLimit` but Monque's API contract is `enqueue(name, data, opts)` — adding per-job limits changes the options interface. Global limit is sufficient for hardening. | Global `maxPayloadSize` in `MonqueOptions` only. Per-job can be a future enhancement.            |
| **Persistent worker registry / heartbeat document**           | BullMQ uses Redis `CLIENT LIST` for worker discovery; pg-boss uses internal DB tables. Adding a persistent registry is a new feature, not hardening.                                          | Simple startup query for instance ID collision. No persistent lock document.                     |
| **Stats export / Prometheus integration**                     | BullMQ has `exportPrometheusMetrics()`. Valuable but scope-creep for a hardening milestone.                                                                                                   | TTL-cached `getQueueStats()` is sufficient. Prometheus export can be a future feature milestone. |
| **Metrics time-series collection**                            | BullMQ stores per-minute data points for completed/failed jobs over configurable windows (MetricsTime). This is a significant new feature.                                                    | Stick with point-in-time stats via `getQueueStats()`.                                            |
| **Dead-letter queue**                                         | Explicitly listed as out of scope in PROJECT.md. Separate feature milestone.                                                                                                                  | Document the pattern for consumers to implement their own via `job:fail` events.                 |
| **Job priority system**                                       | Explicitly listed as out of scope in PROJECT.md. Separate feature milestone.                                                                                                                  | No action — document as future work.                                                             |
| **Job progress tracking**                                     | Explicitly listed as out of scope in PROJECT.md. BullMQ has `updateProgress()`.                                                                                                               | No action — future feature milestone.                                                            |
| **Job timeout enforcement**                                   | Explicitly listed as out of scope in PROJECT.md.                                                                                                                                              | No action — separate from heartbeats which already exist.                                        |
| **Deprecated option removal**                                 | `defaultConcurrency`/`maxConcurrency` removal requires major version bump.                                                                                                                    | Leave deprecation warnings in place. Remove in next major.                                       |
| **Full facade decomposition (separate Queue/Worker classes)** | BullMQ's architectural split is fundamentally different from Monque's unified facade. Splitting now would be a major API redesign.                                                            | Reduce JSDoc bloat and optionally extract LifecycleManager internally. Keep the unified facade.  |
| **Schema validation middleware / hooks**                      | Graphile Worker recommends application-level validation. Adding middleware/hooks is a new API surface.                                                                                        | `maxPayloadSize` byte check is sufficient. Schema validation is consumer's responsibility.       |

## Feature Dependencies

```
TS-5 (Test Gaps)         → No dependencies (do first, no source changes)
TS-3 (Payload Validation) → No dependencies (new error class + option)
TS-4 (Instance ID Check)  → No dependencies (startup query)
TS-2 (Stats Caching)      → No dependencies (internal to JobQueryService)
TS-1 (Bulk Operations)    → No dependencies (internal to JobManager)
D-2 (Mapper Robustness)   → TS-5 partially (round-trip test informs mapper changes)
D-1 (Class Size)           → TS-1, TS-2, TS-3, TS-4 (refactor after functional changes to avoid merge conflicts)
```

**Key insight:** D-1 (class size reduction) should be last because JSDoc changes and potential LifecycleManager extraction touch the same file (monque.ts) as several other features. Doing it last avoids merge conflict churn.

## Milestone Recommendation

### Priority order (highest first):

1. **TS-5: Test coverage gaps** — Zero source changes, immediate risk reduction, validates existing behavior before modifying anything
2. **TS-3: Payload validation** — Low complexity, addresses security audit concern, new error class is additive
3. **TS-4: Instance ID collision check** — Low complexity, addresses security audit concern, single startup query
4. **TS-2: Stats caching** — Low complexity, measurable performance improvement, self-contained in JobQueryService
5. **TS-1: Bulk operation optimization** — Medium complexity, measurable performance improvement (O(n) → O(1)), most impactful for large queues
6. **D-2: Mapper robustness** — Medium complexity, prevents future regressions, test + spread-based approach
7. **D-1: Class size reduction** — Medium complexity, pure maintenance improvement, do last to avoid conflicts

### Deferred:

- **Prometheus/metrics export** — future feature milestone
- **Per-job sizeLimit** — future enhancement after global limit proves useful
- **Persistent worker registry** — future feature if multi-instance observability becomes a priority
